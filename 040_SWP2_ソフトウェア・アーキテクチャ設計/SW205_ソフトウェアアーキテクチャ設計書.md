# **概要**

## **目的**
本ソフトウェアアーキテクチャ設計書は、「スマートグリーンハウスIoTシステム」のソフトウェアコンポーネントの構造、相互関係、および主要な設計決定を明確に記述することを目的とします。これにより、ソフトウェア開発のガイドラインを提供し、システムの拡張性、保守性、信頼性を確保します。

## **位置づけ**
本ソフトウェアアーキテクチャ設計書は、システム要求仕様書 (SY106) およびソフトウェア要求仕様書 (SW105) の要件に基づき、システムアーキテクチャ設計書 (SY205) の上位設計を受け、ソフトウェア開発チームが実装を行うための基礎となります。

## **対象ユーザ**
*   ソフトウェア開発者
*   システムアーキテクト
*   テストエンジニア

## **記載範囲**
本設計書は、以下の主要なソフトウェアコンポーネントを対象とします。
*   エッジデバイスファームウェア
*   クラウドバックエンドシステム
*   クラウドフロントエンドシステム (Web/モバイルアプリケーション)

# **4 制御方式**

## **4.1 メモリー構成／レイアウト**

*   **エッジデバイス**:
    *   **フラッシュメモリ**: ファームウェアコード、設定値、キャリブレーションデータなどを格納。OTA (Over-The-Air) アップデート領域を確保。
    *   **RAM**: 実行時データ、センサーデータバッファ、通信バッファ、スタック、ヒープ。
    *   **EEPROM/FRAM**: 永続化が必要な設定値（例: デバイスID、Wi-Fiクレデンシャル、制御閾値）を格納。

*   **クラウドバックエンド**: 各マイクロサービスはコンテナ環境で動作し、必要なメモリは動的に割り当てられます。時系列データベースは大量のデータを効率的に扱うために最適化されたメモリ管理を行います。

## **4.2 ソフトウェア制御方式**

### 4.2.1 エッジデバイスの制御フロー (メインループ)

```mermaid
flowchart TD
    A[起動/初期化] --> B{センサーデータ収集};
    B --> C{制御ロジック実行};
    C --> D{通信処理 (クラウドとの同期)};
    D --> B;
```

*   **初期化**: 各モジュール (HAL, センサー, アクチュエータ, 通信) の初期設定を行います。
*   **センサーデータ収集**: 定期的に全センサーからデータを読み取り、整形し、データバッファに格納します。
*   **制御ロジック実行**: データバッファ内の最新データと設定値に基づき、環境制御アルゴリズムを実行し、必要に応じてアクチュエータを制御します。手動制御コマンドがある場合はそちらを優先します。
*   **通信処理**: 一定周期でクラウドバックエンドにセンサーデータを送信し、クラウドからの制御コマンドや設定値を受信します。

### 4.2.2 クラウドバックエンドの制御方式

*   **非同期メッセージング**: エッジデバイスからのデータインジェスト、データ処理、通知生成などの処理は、メッセージキュー (Kafka/RabbitMQなど) を介した非同期処理を基本とします。これにより、システムの高いスケーラビリティと耐障害性を実現します。
*   **API駆動**: フロントエンドからのリクエストやAIモジュールからの指示は、RESTful APIを介して各サービスにルーティングされます。認証・認可はAPIゲートウェイで一元的に行われます。
*   **イベント駆動**: 特定のイベント（例: 閾値超過、デバイス接続断）発生時に、関連するサービス（例: 通知サービス、AI連携サービス）がトリガーされるイベント駆動アーキテクチャを採用します。

## **4.3 性能見積**

*   **エッジデバイス**:
    *   **センサーデータ収集頻度**: 1秒間に1回のデータ収集と処理を目標とします。
    *   **通信頻度**: 1分間に1回のクラウドへのデータ送信を目標とします。LoRaWANの場合は10分に1回程度。
    *   **バッテリー寿命**: 最低1年間（ソーラーパネル併用で無制限）を目標とします。

*   **クラウドバックエンド**:
    *   **データインジェスト**: 1秒あたり1000デバイスからの同時データ受信を処理可能とします。
    *   **API応答速度**: ユーザー向けAPIは平均100ms以内、最大500ms以内を目標とします。
    *   **データ処理**: 異常検知、集計処理はデータ受信後5秒以内に完了を目標とします。


# **5 機能ユニット詳細**

### 5.1 エッジデバイスファームウェア

#### 5.1.1 構成要素
*   **ハードウェア抽象化層 (HAL)**: センサー、アクチュエータ、通信モジュールなどのハードウェアリソースを抽象化し、上位層に共通インターフェースを提供します。
*   **センサー管理モジュール**: 各種センサーからデータを周期的に取得し、生データを整形します。
*   **アクチュエータ制御モジュール**: 制御コマンドに応じて、対応するアクチュエータを操作します。安全確保のため、排他制御やフェイルセーフ機構を考慮します。
*   **通信モジュール**: LoRaWANまたはWi-Fiモジュールを介してクラウドバックエンドとのデータ送受信を担います。再送処理やエラーハンドリングを実装します。
*   **制御ロジックモジュール**: 収集されたセンサーデータとクラウドからの設定値に基づき、環境制御の判断を行います。通信途絶時には、直前の設定値に基づいた自律制御に切り替わります。
*   **データバッファモジュール**: センサーデータや制御設定値を一時的に保持します。通信障害時の一時的なデータ保持にも利用します。

#### 5.1.2 データフロー (エッジデバイス内)

```mermaid
sequenceDiagram
    participant Sensor
    participant SensorManager[Edge]
    participant ControlLogic[Edge]
    participant ActuatorManager[Edge]
    participant Actuator
    participant CommunicationManager[Edge]
    participant CloudBackend

    Sensor->>SensorManager: データ計測
    SensorManager->>ControlLogic: センサーデータ提供
    ControlLogic->>ActuatorManager: 制御指示
    ActuatorManager->>Actuator: アクチュエータ操作
    SensorManager->>CommunicationManager: センサーデータ送信準備
    CommunicationManager->>CloudBackend: センサーデータ送信
    CloudBackend->>CommunicationManager: 制御コマンド受信
    CommunicationManager->>ControlLogic: 制御コマンド伝達
```

### 5.2 クラウドバックエンドシステム

#### 5.2.1 構成要素
*   **APIゲートウェイ**: 全ての外部からのAPIリクエストを受け付け、認証・認可を行い、適切なバックエンドサービスにルーティングします。
*   **データインジェストサービス**: エッジデバイスからのセンサーデータを受け取り、初期検証と前処理を行い、時系列データベースに書き込みます。デバイス認証も担当します。
*   **時系列データベース**: センサーデータ、アクチュエータ履歴、制御履歴などを効率的に格納・検索するためのデータベースです。
*   **データ処理サービス**: 蓄積されたデータに対し、異常検知、集計、統計分析などを実行します。AI連携のためのデータ整形も行います。

*   **ハードウェア抽象化層 (HAL)**: センサー、アクチュエータ、通信モジュールなどのハードウェアリソースを抽象化し、上位層に共通インターフェースを提供します。

*   **データバッファモジュール**: センサーデータや制御設定値を一時的に保持します。通信障害時の一時的なデータ保持にも利用します。

### 4. 主要な設計上の決定事項

#### 4.1 通信プロトコル
*   **エッジ - クラウド間**: MQTT/Sを主とし、低帯域・非同期通信の要件を満たします。LoRaWANの場合はCoAPや軽量なカスタムプロトコルも検討します。信頼性向上のためQoSレベルを考慮します。
*   **クラウド - フロントエンド間**: RESTful API (HTTP/S) を標準とします。リアルタイム性が求められるダッシュボード更新にはWebSocketの利用も検討します。

#### 4.2 データ永続化
*   **センサーデータ**: 時系列データベース（例: TimescaleDB, InfluxDB）を採用し、大量の時系列データを効率的に保存・クエリできるようにします。
*   **設定・マスターデータ**: リレーショナルデータベース（例: PostgreSQL）またはNoSQLデータベース（例: DynamoDB, MongoDB）を採用し、ユーザー情報、デバイス情報、閾値設定などを管理します。

#### 4.3 認証・認可
*   **デバイス認証**: X.509証明書またはShared Key方式を用いて、各エッジデバイスの正当性を検証します。
*   **ユーザー認証**: JWT (JSON Web Token) ベースの認証メカニズムを採用し、ステートレスな認証を可能にします。OAuth 2.0フローを検討します。
*   **認可**: ロールベースアクセス制御 (RBAC) を導入し、ユーザーの役割に応じてアクセス可能な機能やデータ範囲を制限します。
